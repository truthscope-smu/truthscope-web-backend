# BDD — URL 입력 어댑터 (사용자가 준 뉴스 URL을 분석 가능한 "기사 카드"로 만드는 부품)

> **작성일**: 2026-05-10
> **이슈**: BE #24 정리 PR
> **읽는 법**: 각 시나리오 위에 "왜 필요한지 / 이 부품에 무엇을 요구하는지" 주석이 먼저 옵니다. 다섯 시나리오를 순서대로 읽으면 부품 설계가 자연스럽게 그려집니다. 마지막 섹션의 "종합 설계도"가 모든 시나리오를 합쳤을 때의 부품 동작 흐름입니다.

---

## 한 줄 요약

사용자가 붙여넣은 뉴스 URL을 받아서, 분석에 쓸 수 있는 **"기사 카드"** 한 장으로 만들어 돌려준다.

여기서 **"기사 카드"** = 우리 시스템이 뉴스 한 건을 다룰 때 쓰는 통일된 묶음 (원본 주소 / 제목 / 본문 / 언어 / 도메인 / 추출 시각의 6개 칸).

---

## 시나리오 1 — 정상 흐름

### 왜 이 시나리오가 필요한가
가장 흔한 사용 사례. 부품의 **기본 동작**을 정의하는 시작점이다. 다른 시나리오들은 이 정상 흐름의 분기·실패 경로다.

### 이 시나리오가 부품에 요구하는 것
1. 입력받은 URL 형식이 맞는지 먼저 확인한다 (외부 요청 전에)
2. 외부 뉴스 서버로 안전하게 요청을 보내 본문을 가져온다 (사설망 차단 같은 보안 점검 포함)
3. 가져온 HTML에서 제목·본문·언어·도메인 정보를 추출한다
4. 추출한 정보 6개 칸을 채워서 "기사 카드" 한 장으로 묶어 돌려준다

### 시나리오

```
주어진 상황:
  사용자가 정상적인 뉴스 URL "https://news.example.com/article/12345"를 입력했다
  외부 뉴스 사이트는 정상 응답을 돌려준다 (제목·본문·언어·도메인이 모두 추출 가능한 상태)

이 부품이 하는 일:
  사용자 URL을 받아 분석에 쓸 기사 카드 한 장을 만든다

결과:
  - 기사 카드가 만들어져 돌려받는다
  - 카드의 "원본 주소" 칸에는 사용자가 입력한 URL이 그대로 들어 있다
  - 카드의 "제목" 칸에는 외부 사이트에서 뽑은 기사 제목이 들어 있다
  - 카드의 "원본 주소"는 비어 있지 않고 http 또는 https로 시작한다 (이 규칙은 카드의 최소 요건이다)
  - 카드에 "추출 시각"이 기록되어 있다
```

## 시나리오 2 — 잘못된 형식의 주소를 거부

### 왜 이 시나리오가 필요한가
사용자가 실수로 또는 악의로 뉴스가 아닌 주소(파일 다운로드 주소, 내부 프로토콜 등)를 입력할 수 있다. 이런 입력은 외부 요청을 시작하기 **전에** 거부해야 한다. 그렇지 않으면 보안 문제가 생기거나 부품이 무의미한 작업을 한다.

### 이 시나리오가 부품에 요구하는 것
1. URL 형식 검사를 시나리오 1의 본문 추출보다 **먼저** 한다
2. 형식이 잘못되면 즉시 "잘못된 입력" 신호를 보내고 동작을 멈춘다
3. 기사 카드는 만들지 않는다

### 시나리오

```
주어진 상황:
  사용자가 뉴스가 아닌 주소 "ftp://malicious.example.com/file.txt"를 입력했다

이 부품이 하는 일:
  사용자 URL을 받아 기사 카드를 만들려고 시도한다

결과:
  - "잘못된 입력" 신호를 보내며 동작을 멈춘다 (이 신호는 호출자가 사용자에게 "URL 형식이 잘못되었습니다"라고 안내할 수 있는 종류여야 한다)
  - 외부 뉴스 서버로의 요청은 보내지지 않는다
  - 기사 카드는 만들어지지 않는다
```

---

## 시나리오 3 — 내부 네트워크 주소를 차단

### 왜 이 시나리오가 필요한가
공격자가 외부에서 우리 서버를 통해 우리 회사 내부망(예: 사내 서버, 클라우드 메타데이터 주소 169.254.169.254)에 접근하려고 시도할 수 있다. 이런 공격을 SSRF(Server-Side Request Forgery)라고 하며, 부품은 외부 요청 전에 이런 주소를 차단해야 한다.

### 이 시나리오가 부품에 요구하는 것
1. URL 형식이 정상이라도 **목적지 IP가 사설/내부 대역**인지 추가 점검한다
2. 사설/내부 IP라면 즉시 "내부망 차단" 신호를 보내고 멈춘다
3. 외부 HTTP 요청은 절대 보내지지 않는다 (요청을 보낸 후 차단하는 게 아니라, **요청 전에** 차단해야 한다)

### 시나리오

```
주어진 상황:
  사용자가 사설 IP를 포함한 URL "http://192.168.1.1/admin"을 입력했다 (192.168.x.x 대역은 회사 내부망에서 흔히 쓰이는 사설 IP다)

이 부품이 하는 일:
  사용자 URL을 받아 기사 카드를 만들려고 시도한다

결과:
  - "내부 네트워크 주소는 차단되었다"는 신호를 보내며 동작을 멈춘다 (이 신호는 시나리오 2의 "잘못된 형식" 신호와는 종류가 달라야 한다 — 형식은 맞지만 보안 정책상 거부된 것이므로)
  - 외부로 단 한 번의 HTTP 요청도 나가지 않는다
  - 기사 카드는 만들어지지 않는다
```

## 시나리오 4 — 외부 서버 응답 실패

### 왜 이 시나리오가 필요한가
입력은 정상이고 보안 검사도 통과했는데, 외부 뉴스 서버 자체가 다운되었거나 5xx 오류를 돌려보내는 경우가 있다. 이는 부품의 잘못이 아니지만, 부품은 이 상황을 적절히 호출자에게 전달해야 한다 (조용히 빈 카드를 돌려주면 안 된다).

### 이 시나리오가 부품에 요구하는 것
1. 외부 요청 단계에서 발생한 실패는 **숨기지 말고** 명확한 신호로 호출자에게 전달한다
2. 신호의 종류는 시나리오 2(형식 오류) / 시나리오 3(보안 차단)과는 또 달라야 한다 — "외부 서버에서 가져오기 실패"라는 별도 신호여야 한다
3. 기사 카드는 만들지 않는다 (반쯤 채워진 카드를 돌려주면 안 된다)

### 시나리오

```
주어진 상황:
  사용자가 정상 URL "https://unavailable.example.com/article/99"를 입력했다
  외부 뉴스 서버는 응답하지 않거나 5xx 오류를 돌려준다

이 부품이 하는 일:
  사용자 URL을 받아 기사 카드를 만들려고 시도한다

결과:
  - "외부 서버에서 기사를 가져올 수 없다"는 신호를 보내며 동작을 멈춘다
  - 기사 카드는 만들어지지 않는다
```

---

## 시나리오 5 — 입력 자체가 비어 있음

### 왜 이 시나리오가 필요한가
호출자가 URL 칸을 비워두거나 공백만 넣어서 부품을 호출할 수 있다. 이런 입력은 시나리오 2(잘못된 형식)와도 다르다 — 형식 검사조차 의미 없는 빈 입력이므로 가장 먼저 거부되어야 한다.

### 이 시나리오가 부품에 요구하는 것
1. URL 입력이 비어 있는지(`null` 또는 공백)를 **가장 먼저** 확인한다
2. 비어 있으면 "잘못된 입력" 신호를 보낸다 (시나리오 2와 같은 종류의 신호)
3. 그 어떤 후속 작업도 시작하지 않는다

### 시나리오

```
주어진 상황:
  호출자가 URL 칸을 비워두고 부품을 호출했다 (값이 없거나, 공백만 있는 경우)

이 부품이 하는 일:
  사용자 URL을 받아 기사 카드를 만들려고 시도한다

결과:
  - "잘못된 입력" 신호를 보내며 즉시 동작을 멈춘다
  - 후속 작업(형식 검사, 외부 요청, 본문 추출 중 어떤 것도)은 시작되지 않는다
  - 기사 카드는 만들어지지 않는다
```

## 종합 설계도 (시나리오 1~5를 합치면 부품은 이렇게 동작한다)

```
[입력] 사용자가 준 URL 문자열
   │
   ▼
1단계: URL이 비어 있는가? (시나리오 5)
   │ 비어 있음 → "잘못된 입력" 신호 → 종료
   │ 비어 있지 않음 → 다음 단계
   ▼
2단계: URL 형식이 http(s)로 시작하는가? (시나리오 2)
   │ 아님 → "잘못된 입력" 신호 → 종료
   │ 맞음 → 다음 단계
   ▼
3단계: URL의 목적지 IP가 사설/내부 대역인가? (시나리오 3)
   │ 사설망 → "내부망 차단" 신호 → 종료
   │ 외부망 → 다음 단계
   ▼
4단계: 외부 뉴스 서버에 요청을 보내 HTML을 받아온다 (시나리오 4)
   │ 서버 오류/타임아웃 → "외부 서버에서 가져오기 실패" 신호 → 종료
   │ 정상 응답 → 다음 단계
   ▼
5단계: HTML에서 제목·본문·언어·도메인을 뽑아낸다 (시나리오 1)
   │
   ▼
6단계: 6개 칸(원본 주소 / 제목 / 본문 / 언어 / 도메인 / 추출 시각)을 채워 기사 카드를 만든다 (시나리오 1)
   │
   ▼
[출력] 기사 카드 1장
```

## 신호의 종류 정리 (호출자가 사용자에게 다른 메시지를 보여주려면 신호가 구분되어야 한다)

| 시나리오 | 신호 이름 | 사용자에게 보여줄 메시지 예시 |
|---------|---------|---------------------------|
| 1 | (정상 — 신호 없음, 카드 반환) | "기사를 분석합니다" |
| 2 | "잘못된 입력" | "URL 형식이 잘못되었습니다. http:// 또는 https://로 시작해야 합니다" |
| 3 | "내부망 차단" | "내부 네트워크 주소는 분석할 수 없습니다" |
| 4 | "외부 서버에서 가져오기 실패" | "기사를 가져올 수 없습니다. 잠시 후 다시 시도해주세요" |
| 5 | "잘못된 입력" (시나리오 2와 동일) | "URL을 입력해주세요" |

## 도메인 용어 정리

| 용어 | 의미 |
|------|------|
| 기사 카드 | 우리 시스템이 뉴스 한 건을 다룰 때 쓰는 통일된 묶음 (원본 주소 / 제목 / 본문 / 언어 / 도메인 / 추출 시각 6개 칸) |
| 카드의 최소 요건 | 카드는 항상 "원본 주소가 있고 http 또는 https로 시작한다"는 규칙을 만족해야 한다. 이 규칙을 어긴 카드는 만들어지면 안 된다 |
| 사설 IP / 내부망 | 회사 내부 네트워크에서 쓰이는 주소 대역 (192.168.x.x, 10.x.x.x, 127.x.x.x, 169.254.x.x 등) |
| SSRF | 외부 공격자가 우리 서버를 통해 내부망에 접근하려는 공격. 시나리오 3이 이 공격을 막는다 |
| 신호 | 부품이 호출자에게 결과를 알리는 방법. 정상이면 카드를 돌려주고, 실패면 종류별로 다른 신호를 보낸다 |
