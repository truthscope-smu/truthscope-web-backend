# BDD — 본문 입력 어댑터 (사용자가 직접 붙여넣은 본문을 분석 가능한 "기사 카드"로 만드는 부품)

> **작성일**: 2026-05-10
> **이슈**: BE #24 정리 PR
> **읽는 법**: 각 시나리오 위에 "왜 필요한지 / 이 부품에 무엇을 요구하는지" 주석이 먼저 옵니다. 다섯 시나리오를 순서대로 읽으면 부품 설계가 자연스럽게 그려집니다. 마지막 섹션의 "종합 설계도"가 모든 시나리오를 합쳤을 때의 부품 동작 흐름입니다.

---

## 한 줄 요약

사용자가 URL이 아니라 **본문 자체를 직접 붙여넣었을 때**, 그 본문을 분석에 쓸 수 있는 **"기사 카드"** 한 장으로 만들어 돌려준다.

URL 입력 어댑터(BDD-url-input-adapter.md)와 같은 형식의 카드를 돌려주지만, 출처가 외부 사이트가 아니라 사용자 직접 입력이라는 점이 다르다.

---

## 왜 이 부품이 따로 필요한가 (URL 어댑터와의 관계)

URL 입력 어댑터는 외부 사이트를 fetch해서 카드를 만든다. 하지만 다음 두 경우에는 URL이 없거나 못 쓴다:

1. 사용자가 **URL 차단된 환경**(예: 회사 방화벽 안)에서 본문만 복사해 왔을 때
2. **이미 검증하고 싶은 의심 본문**을 외부 사이트가 아닌 메신저·이메일 등에서 받아왔을 때

이때 본문 입력 어댑터가 그 raw text를 받아 같은 "기사 카드"로 만들어주면, 후속 분석 파이프라인은 카드의 출처가 URL이든 본문이든 신경 쓰지 않고 똑같이 처리할 수 있다.

---

## 사전 결정 (시나리오를 읽기 전에 알아둘 것)

기사 카드의 "원본 주소" 칸은 원래 http(s) URL만 받게 되어 있다 (시스템 보안·일관성을 위해). 본문 입력의 경우 진짜 URL이 없는데, 이를 어떻게 처리할지 세 가지 선택지가 있다:

- **선택지 A**: 가짜 URL(예: `text://input`)을 채워 넣는다 → **기존 규칙과 충돌** (http로 시작 안 함)
- **선택지 B**: "본문에서 만든 카드"용 별도 생성 통로를 카드 모델에 추가한다 → 원본 주소 칸은 비워두고, 본문 출처임을 카드 자체가 알게 한다
- **선택지 C**: 본문 입력 어댑터는 카드를 안 만들고 부품 절반(중간 데이터)만 돌려주고 호출자가 카드를 만든다 → 이슈 요구사항(어댑터가 카드를 만든다)과 충돌

**본 BDD는 선택지 B를 가정**한다. 즉 카드 모델에 "본문에서 만든 카드용 새 통로"가 추가되며, 이 통로를 쓰면 원본 주소 칸이 비어 있어도 카드가 유효하다. 이 결정은 다음 단계 codex 평가에서 다시 검토할 예정.

## 시나리오 1 — 정상 흐름 (여러 줄 본문)

### 왜 이 시나리오가 필요한가
사용자가 보통 입력하는 형태. 신문 기사를 복사해서 붙여넣으면 첫 줄은 보통 제목, 나머지는 본문이다. 이 일반 사용 사례가 부품의 **기본 동작**을 정의한다.

### 이 시나리오가 부품에 요구하는 것
1. 입력 본문이 비어 있지 않은지 먼저 확인한다
2. 첫 줄을 잘라서 카드의 "제목" 칸에 넣는다
3. 나머지 줄들을 모아서 카드의 "본문" 칸에 넣는다
4. 언어 칸은 한국어로 고정 (이후 단계에서 자동 감지로 발전 예정)
5. 6개 칸을 채워 카드 한 장을 돌려준다 (원본 주소 칸은 비어 있어도 됨 — 사전 결정 선택지 B)

### 시나리오

```
주어진 상황:
  사용자가 신문 기사를 복사해서 붙여넣었다.
  여러 줄로 되어 있고 첫 줄은 제목 같은 형태다.
  예) "정부, 2026년 경제성장률 2.1% 전망 발표
       한국 정부는 오늘 기획재정부를 통해 올해 경제성장률 전망치를 2.1%로 발표했다.
       전문가들은 수출 회복세가 주요 요인이라고 분석했다."

이 부품이 하는 일:
  붙여넣은 본문을 받아 기사 카드 한 장을 만든다

결과:
  - 카드가 만들어져 돌려받는다
  - 카드의 "제목" 칸에는 첫 줄("정부, 2026년 경제성장률 2.1% 전망 발표")이 들어 있다
  - 카드의 "본문" 칸에는 나머지 줄들이 모두 들어 있다
  - 카드의 "언어" 칸에는 "ko"가 들어 있다
  - 카드에 "추출 시각"이 기록되어 있다
  - 카드의 "원본 주소" 칸은 비어 있다 (외부 URL이 없으므로 정상)
```

## 시나리오 2 — 입력이 아예 없을 때 거부

### 왜 이 시나리오가 필요한가
호출자가 본문 칸을 비워두고 부품을 호출할 수 있다. 빈 입력으로는 카드를 만들 수 없으므로 가장 먼저 거부해야 한다.

### 이 시나리오가 부품에 요구하는 것
1. 본문 입력이 `null`인지 가장 먼저 확인한다
2. `null`이면 "잘못된 입력" 신호를 보내고 즉시 멈춘다
3. 이후 작업(첫 줄 자르기, 카드 만들기)은 시작도 하지 않는다

### 시나리오

```
주어진 상황:
  호출자가 본문 칸에 아무 값도 넣지 않고 부품을 호출했다 (값 자체가 없는 상태)

이 부품이 하는 일:
  본문을 받아 기사 카드를 만들려고 시도한다

결과:
  - "잘못된 입력" 신호를 보내며 즉시 동작을 멈춘다
  - 어떤 후속 작업도 시작되지 않는다
  - 카드는 만들어지지 않는다
```

---

## 시나리오 3 — 입력은 있지만 공백만 있을 때 거부

### 왜 이 시나리오가 필요한가
호출자가 빈칸 또는 줄바꿈만 입력해서 보낼 수도 있다. 이건 시나리오 2의 "값이 없음"과는 다르지만, 결과적으로 의미 있는 본문이 없다는 점에서 똑같이 거부해야 한다.

### 이 시나리오가 부품에 요구하는 것
1. 본문이 `null`이 아니더라도 **앞뒤 공백을 제거하고 나서** 빈 문자열인지 확인한다
2. 빈 문자열이면 시나리오 2와 같은 종류의 "잘못된 입력" 신호를 보낸다
3. 카드는 만들지 않는다

### 시나리오

```
주어진 상황:
  사용자가 공백 문자(스페이스, 탭, 줄바꿈)만 입력했다
  예) "   \t\n  "

이 부품이 하는 일:
  본문을 받아 기사 카드를 만들려고 시도한다

결과:
  - "잘못된 입력" 신호를 보내며 동작을 멈춘다 (시나리오 2와 같은 종류의 신호)
  - 카드는 만들어지지 않는다
```

---

## 시나리오 4 — 한 줄만 입력했을 때

### 왜 이 시나리오가 필요한가
사용자가 짧은 사실만 검증하려고 한 줄짜리 문장을 입력할 수 있다 (예: "정부가 2.1% 경제성장률을 발표했다"). 이 경우 본문은 비지만, 제목은 있다. 이 입력을 거부할지 받아들일지 결정이 필요하다.

**정책 결정**: 받아들인다. 제목만 있는 카드도 분석 대상으로 의미 있다 (Tier 2 검증 단계가 제목 한 줄도 처리할 수 있다).

### 이 시나리오가 부품에 요구하는 것
1. 한 줄짜리 입력도 거부하지 않는다
2. 첫 줄 = 제목 칸, 본문 칸은 빈 문자열로 채운다
3. 나머지는 시나리오 1과 동일

### 시나리오

```
주어진 상황:
  사용자가 줄바꿈 없이 한 줄만 입력했다
  예) "한국 경제 성장률 발표"

이 부품이 하는 일:
  본문을 받아 기사 카드 한 장을 만든다

결과:
  - 카드가 만들어져 돌려받는다
  - 카드의 "제목" 칸에는 입력한 한 줄이 들어 있다
  - 카드의 "본문" 칸에는 빈 문자열이 들어 있다
  - 카드의 "언어" 칸에는 "ko"가 들어 있다
```

---

## 시나리오 5 — 앞뒤 공백 정리

### 왜 이 시나리오가 필요한가
사용자가 본문을 복사할 때 앞뒤에 공백이나 줄바꿈이 묻어올 수 있다 (특히 모바일 메신저에서 복사할 때 자주 발생). 이걸 그대로 카드에 담으면 분석 결과의 품질이 떨어진다.

### 이 시나리오가 부품에 요구하는 것
1. 입력 전체에서 앞뒤 공백을 제거한 다음에 첫 줄 자르기를 한다
2. 잘라낸 제목과 본문에서도 각각 앞뒤 공백을 제거한다

### 시나리오

```
주어진 상황:
  사용자가 입력한 본문의 앞뒤에 불필요한 공백이 묻어 있다
  예) "  제목 줄
        본문 줄  "

이 부품이 하는 일:
  본문을 받아 기사 카드 한 장을 만든다

결과:
  - 카드가 만들어져 돌려받는다
  - 카드의 "제목" 칸에는 "제목 줄" (앞뒤 공백 제거됨)
  - 카드의 "본문" 칸에는 "본문 줄" (앞뒤 공백 제거됨)
```

## 종합 설계도 (시나리오 1~5를 합치면 부품은 이렇게 동작한다)

```
[입력] 사용자가 붙여넣은 본문 문자열
   │
   ▼
1단계: 본문이 null인가? (시나리오 2)
   │ null → "잘못된 입력" 신호 → 종료
   │ null 아님 → 다음 단계
   ▼
2단계: 앞뒤 공백을 제거한 본문이 빈 문자열인가? (시나리오 3)
   │ 빈 문자열 → "잘못된 입력" 신호 → 종료
   │ 빈 문자열 아님 → 다음 단계
   ▼
3단계: 본문을 첫 줄 / 나머지 줄로 나눈다 (시나리오 1, 4, 5)
   │ 줄바꿈 있음 → 첫 줄 = 제목, 나머지 = 본문
   │ 줄바꿈 없음 → 전체 = 제목, 본문 = 빈 문자열 (시나리오 4)
   ▼
4단계: 제목과 본문 각각의 앞뒤 공백을 제거한다 (시나리오 5)
   │
   ▼
5단계: 카드의 6개 칸을 채운다
   - 원본 주소: 비어 있음 (외부 URL이 없으므로 정상)
   - 제목: 3단계에서 만든 제목
   - 본문: 3단계에서 만든 본문
   - 언어: "ko" 고정 (이후 자동 감지로 발전)
   - 도메인: "본문 직접 입력" 같은 출처 표시
   - 추출 시각: 현재 시각
   ▼
[출력] 기사 카드 1장
```

## 신호의 종류 정리

| 시나리오 | 신호 이름 | 사용자에게 보여줄 메시지 예시 |
|---------|---------|---------------------------|
| 1 | (정상 — 카드 반환) | "본문을 분석합니다" |
| 2 | "잘못된 입력" | "본문을 입력해주세요" |
| 3 | "잘못된 입력" (시나리오 2와 동일) | "본문을 입력해주세요" |
| 4 | (정상 — 카드 반환) | "본문을 분석합니다" |
| 5 | (정상 — 카드 반환) | "본문을 분석합니다" |

## 도메인 용어 정리

| 용어 | 의미 |
|------|------|
| 기사 카드 | URL 어댑터와 동일. 6개 칸(원본 주소 / 제목 / 본문 / 언어 / 도메인 / 추출 시각)으로 구성 |
| 본문에서 만든 카드 | 외부 URL 없이 사용자가 직접 입력한 본문으로 만든 카드. 원본 주소 칸이 비어 있어도 유효함 |
| 신호 | 부품이 호출자에게 결과를 알리는 방법. 정상이면 카드를 돌려주고, 실패면 "잘못된 입력" 신호를 보낸다 |
| 한국어 고정 | 현재(MVP) 단계에서는 모든 입력을 "ko"로 처리. 이후 단계에서 자동 감지 라이브러리 도입 예정 |
